# Clear Progress

**GTD-приложение для доведения дел до конца**

## BACKEND API

**Google Apps Script Web App**

Версия 1.0 | Март 2026

---

## 1. Общие принципы

### 1.1. Архитектура

Backend реализован как Google Apps Script Web App. Каждый пользователь разворачивает собственный экземпляр в своём Google-аккаунте. API доступен по URL вида:

```
https://script.google.com/macros/s/{DEPLOY_ID}/exec
```

Apps Script поддерживает два HTTP-метода: `doGet` (GET-запросы) и `doPost` (POST-запросы с JSON-телом). Маршрутизация осуществляется через поле `action` в теле запроса (POST) или query-параметр (GET).

### 1.2. Подход к API

API построен по sync-ориентированной модели с отдельными эндпоинтами для бинарных операций. Клиент (PWA) работает с локальной базой IndexedDB и периодически синхронизируется с сервером пакетно. Это минимизирует количество HTTP-запросов (критично для Apps Script с холодным стартом ~1–2 сек) и обеспечивает офлайн-работу.

### 1.3. Формат обмена

- Все запросы и ответы в формате JSON (`Content-Type: application/json`)
- Даты в формате ISO 8601 UTC (`2026-03-04T12:00:00.000Z`)
- Идентификаторы в формате UUID v4
- Ответ всегда содержит поле `ok` (boolean) и опционально `error` (string)

### 1.4. Разрешение конфликтов

Применяется стратегия **last-write-wins**: при пуше сервер сравнивает `updated_at` клиентской записи с серверной. Если клиентская новее — принимается. Если серверная новее — клиентская игнорируется, серверная версия возвращается клиенту в ответе push. Для персонального приложения реальные конфликты крайне редки.

### 1.5. Безопасность

Web App деплоится с настройкой «Execute as: Me» и «Who has access: Anyone». Это позволяет PWA вызывать API без OAuth-флоу в браузере. Защита обеспечивается тем, что URL деплоя содержит уникальный идентификатор и известен только владельцу. OAuth-скоупы ограничены: `drive.file` (только созданные скриптом файлы) и `spreadsheets`.

## 3. GET ping

Проверка доступности Web App. Используется при подключении бэкенда для проверки, что URL валиден и скрипт отвечает.

**Request**

```
GET {WEB_APP_URL}?action=ping
```

**Response (200)**

```json
{
  "ok": true,
  "app": "clear_progress",
  "version": "1.0",
  "initialized": true | false
}
```

Поле `initialized` показывает, создана ли уже структура данных (Sheets + папка). Если `false` — клиент должен вызвать `init`.

## 4. POST init

Первоначальная настройка: создание папки `Clear Progress`, файла Google Sheets с 6 листами, подпапки `Covers`. Идемпотентен: повторный вызов не создаёт дубликатов (проверяет PropertiesService).

**Request**

```
POST {WEB_APP_URL}
```

```json
{
  "action": "init"
}
```

**Response (200)**

```json
{
  "ok": true,
  "created": true | false,
  "spreadsheet_id": "1abc...xyz",
  "folder_id": "0Bxyz..."
}
```

Поле `created = true` при первом вызове, `false` при повторном (структура уже существует).

### 4.1. Логика init

1. Проверить PropertiesService на наличие сохранённых ID. Если есть — вернуть их (`created: false`).
2. Создать папку «Clear Progress» в корне Google Drive (через Drive Advanced Service, scope `drive.file`).
3. Создать подпапку «Covers» внутри.
4. Создать файл Google Sheets «Clear Progress Data» внутри папки.
5. Создать 6 листов (`Tasks`, `Goals`, `Contexts`, `Categories`, `Checklist_Items`, `Settings`) с заголовками колонок.
6. Записать настройки по умолчанию в лист `Settings` (`default_box: inbox`, `accent_color: green`).
7. Сохранить `spreadsheet_id`, `folder_id`, `covers_folder_id` в PropertiesService.

## 5. POST pull

Клиент отправляет словарь максимальных `version` по каждой сущности. Сервер возвращает все записи, чья `version` больше клиентской. При первом pull (все версии = 0) сервер отдаёт все данные целиком.

**Request**

```
POST {WEB_APP_URL}
```

```json
{
  "action": "pull",
  "versions": {
    "tasks": 0,
    "goals": 0,
    "contexts": 0,
    "categories": 0,
    "checklist_items": 0
  }
}
```

**Request — поля**

| **Поле** | **Тип** | **Обяз.** | **Описание** |
|----------|---------|-----------|--------------|
| `action` | string | Да | Значение: `"pull"` |
| `versions` | object | Да | Словарь: `имя_сущности → максимальная известная клиенту version`. `0` = первый pull. |

**Response (200)**

```json
{
  "ok": true,
  "data": {
    "tasks": [ { ...task }, { ...task } ],
    "goals": [ { ...goal } ],
    "contexts": [ { ...context } ],
    "categories": [ { ...category } ],
    "checklist_items": [ { ...item } ]
  },
  "settings": [
    { "key": "default_box", "value": "inbox", "updated_at": "..." },
    { "key": "accent_color", "value": "green", "updated_at": "..." }
  ],
  "server_time": "2026-03-04T12:00:00.000Z"
}
```

Каждый массив в `data` содержит только записи, чья `version` превышает клиентскую. Включая удалённые (`is_deleted: true`) — чтобы клиент узнал об удалении. `Settings` возвращаются всегда полностью (их мало, дельта не нужна). `server_time` используется для коррекции рассинхронизации часов между клиентом и сервером.

## 6. POST push

Клиент отправляет пачку локальных изменений. Сервер применяет их по правилу last-write-wins и возвращает результат по каждой записи.

**Request**

```
POST {WEB_APP_URL}
```

```json
{
  "action": "push",
  "changes": {
    "tasks": [
      {
        "id": "uuid-1",
        "title": "Купить молоко",
        "box": "today",
        "is_completed": false,
        "is_deleted": false,
        "updated_at": "2026-03-04T12:00:00.000Z",
        "version": 3
      }
    ],
    "goals": [],
    "contexts": [],
    "categories": [],
    "checklist_items": [],
    "settings": [
      { "key": "accent_color", "value": "#FF5722", "updated_at": "..." }
    ]
  }
}
```

**Request — поля**

| **Поле** | **Тип** | **Обяз.** | **Описание** |
|----------|---------|-----------|--------------|
| `action` | string | Да | Значение: `"push"` |
| `changes` | object | Да | Словарь с массивами изменённых записей по каждой сущности. Каждая запись должна содержать ВСЕ поля сущности (полный снимок). |

**Response (200)**

```json
{
  "ok": true,
  "results": {
    "tasks": [
      { "id": "uuid-1", "status": "accepted", "version": 4 },
      { "id": "uuid-2", "status": "conflict", "server_record": { ... } }
    ],
    "goals": [],
    "contexts": [],
    "categories": [],
    "checklist_items": [],
    "settings": [
      { "key": "accent_color", "status": "accepted" }
    ]
  },
  "server_time": "2026-03-04T12:00:01.000Z"
}
```

### 6.1. Статусы результатов

- **`accepted`** — изменение принято, `version` увеличена на сервере
- **`conflict`** — серверная запись новее (`updated_at`). Поле `server_record` содержит актуальную версию — клиент должен перезаписать локальную копию
- **`created`** — новая запись добавлена на сервер (`id` не найден в Sheets)

### 6.2. Логика обработки push

Для каждой записи в `changes`:

1. Найти запись на сервере по `id`.
2. Если не найдена — добавить новую строку, вернуть `status: created`.
3. Если найдена и клиентская `updated_at` > серверной — обновить, увеличить `version`, вернуть `status: accepted`.
4. Если найдена и серверная `updated_at` >= клиентской — отклонить, вернуть `status: conflict` с серверной записью.

## 7. POST upload_cover

Загрузка изображения-обложки цели на Google Drive в подпапку `Covers`. Изображение передаётся в base64. Максимальный размер — 2 МБ (ограничение полезной нагрузки Apps Script).

**Request**

```
POST {WEB_APP_URL}
```

```json
{
  "action": "upload_cover",
  "goal_id": "uuid-goal-1",
  "filename": "cover.jpg",
  "mime_type": "image/jpeg",
  "data": "/9j/4AAQSkZJRg..."
}
```

**Request — поля**

| **Поле** | **Тип** | **Обяз.** | **Описание** |
|----------|---------|-----------|--------------|
| `action` | string | Да | Значение: `"upload_cover"` |
| `goal_id` | string | Да | UUID цели, к которой привязывается обложка. |
| `filename` | string | Да | Имя файла с расширением (`cover.jpg`, `photo.png`). |
| `mime_type` | string | Да | MIME-тип: `image/jpeg`, `image/png`, `image/webp`. |
| `data` | string | Да | Изображение в base64 (без префикса `data:...`). |

**Response (200)**

```json
{
  "ok": true,
  "file_id": "1abc...xyz",
  "thumbnail_url": "https://drive.google.com/thumbnail?id=1abc...xyz&sz=w400",
  "reused": false
}
```

**Response (200) — дубликат**

```json
{
  "ok": true,
  "file_id": "1abc...xyz",
  "thumbnail_url": "https://drive.google.com/thumbnail?id=1abc...xyz&sz=w400",
  "reused": true
}
```

Поле `reused = true` означает, что идентичное изображение уже существовало и было переиспользовано без создания дубликата.

> **Примечание:** поле `thumbnail_url` в ответе устарело (deprecated) и больше не используется клиентом. Обложки отображаются исключительно через API `get_cover` и кеш IndexedDB. Поле сохраняется в ответе для обратной совместимости.

### 7.1. Дедупликация изображений

Сервер вычисляет SHA-256 хеш загружаемого изображения и сравнивает с хешами существующих файлов в папке `Covers`. Хеши хранятся в описании файла (`description`) на Google Drive. Если хеш совпадает — новый файл не создаётся, возвращается `file_id` существующего файла с признаком `reused: true`.

### 7.2. Именование файлов

Файлы обложек именуются по хешу содержимого: `{sha256_prefix_12}.{ext}`. Например: `a3f8b2c1d4e5.jpg`. Первые 12 символов SHA-256 уникально идентифицируют изображение. Полный хеш хранится в `description` файла для точного сравнения. Имя не привязано к конкретной цели, что корректно при дедупликации (один файл может использоваться несколькими целями).

### 7.3. Логика

1. Декодировать base64 в blob.
2. Вычислить SHA-256 хеш изображения (через `Utilities.computeDigest`).
3. Перебрать файлы в папке `Covers` и сравнить хеш из `description` с вычисленным.
4. Если совпадение найдено — вернуть `file_id` существующего файла (`reused: true`). Новый файл не создаётся.
5. Если совпадений нет — создать новый файл с именем `{sha256_prefix_12}.{ext}`, записать полный хеш в `description`.
6. Сделать файл публичным для чтения (чтобы PWA могла загрузить по thumbnail URL).
7. Вернуть `file_id` и `thumbnail_url` (`reused: false`).

## 8. POST delete_cover

Удаление файла обложки с Google Drive. Перед удалением сервер проверяет, не используется ли файл другими целями (дедупликация). Физическое удаление происходит только когда ни одна цель (включая мягко удалённые) больше не ссылается на файл. Вызывается клиентом при смене или снятии обложки, но не при удалении цели.

**Request**

```
POST {WEB_APP_URL}
```

```json
{
  "action": "delete_cover",
  "file_id": "1abc...xyz"
}
```

**Request — поля**

| **Поле** | **Тип** | **Обяз.** | **Описание** |
|----------|---------|-----------|--------------|
| `action` | string | Да | Значение: `"delete_cover"` |
| `file_id` | string | Да | ID файла на Google Drive для удаления. |

**Response (200) — файл удалён**

```json
{
  "ok": true,
  "deleted": true,
  "ref_count": 0
}
```

**Response (200) — файл ещё используется**

```json
{
  "ok": true,
  "deleted": false,
  "ref_count": 1,
  "message": "Файл используется другими целями"
}
```

### 8.1. Логика подсчёта ссылок

1. Прочитать колонку `cover_file_id` из листа `Goals` через `getDataRange()`.
2. Подсчитать, сколько целей (включая `is_deleted = true`) ссылаются на этот `file_id`.
3. Если ссылок = 0 — удалить файл с Google Drive, вернуть `deleted: true`.
4. Если ссылок > 0 — не удалять, вернуть `deleted: false` с количеством оставшихся ссылок.

Подсчёт ссылок выполняется на лету при каждом вызове `delete_cover`. Учитываются все цели, включая мягко удалённые (`is_deleted = true`), чтобы при восстановлении цели обложка не потерялась. Отдельный счётчик не хранится.

## 9. POST get_cover

Получение одного или нескольких файлов обложек с Google Drive. Файлы возвращаются в виде base64-строк. Поддерживает batch-запросы (до 10 файлов за раз).

**Request**

```
POST {WEB_APP_URL}
```

```json
{
  "action": "get_cover",
  "file_ids": ["1abc...xyz", "1def...uvw"]
}
```

**Request — поля**

| **Поле** | **Тип** | **Обяз.** | **Описание** |
|----------|---------|-----------|--------------|
| `action` | string | Да | Значение: `"get_cover"` |
| `file_ids` | string[] | Да | Массив ID файлов на Google Drive. Минимум 1, максимум 10. |

**Response (200) — успех**

```json
{
  "ok": true,
  "covers": [
    {
      "file_id": "1abc...xyz",
      "mime_type": "image/jpeg",
      "data": "/9j/4AAQSkZJRg..."
    }
  ]
}
```

**Response (200) — частичный успех**

Если один из файлов не найден, остальные всё равно возвращаются. Поле `ok` остаётся `true`.

```json
{
  "ok": true,
  "covers": [
    {
      "file_id": "1abc...xyz",
      "mime_type": "image/jpeg",
      "data": "/9j/4AAQSkZJRg..."
    },
    {
      "file_id": "1NONEXISTENT",
      "error": "FILE_NOT_FOUND"
    }
  ]
}
```

**Response — поля массива `covers`**

| **Поле** | **Тип** | **Описание** |
|----------|---------|--------------|
| `file_id` | string | ID файла (всегда присутствует) |
| `mime_type` | string | MIME-тип изображения (при успехе) |
| `data` | string | Данные файла в base64 без префикса `data:...` (при успехе) |
| `error` | string | Код ошибки (`FILE_NOT_FOUND`) если файл недоступен |

### 9.1. Логика

1. Валидировать `file_ids`: массив должен быть непустым и содержать не более 10 элементов.
2. Для каждого `file_id`:
   - Попытаться получить файл через `DriveApp.getFileById(fileId)`.
   - Прочитать байты: `file.getBlob().getBytes()`.
   - Закодировать в base64: `Utilities.base64Encode(bytes)`.
   - При ошибке (файл не найден или удалён) — вернуть `{ file_id, error: "FILE_NOT_FOUND" }`.
3. Вернуть массив результатов. Частичный успех допустим.

### 9.2. Ограничения

- Максимум 10 файлов за запрос (`MAX_COVER_BATCH_SIZE = 10`). Ограничение вызвано лимитом выполнения GAS (6 минут) и размером ответа (~27 МБ при 10 файлах по 2 МБ).
- Доступны только файлы, созданные этим же скриптом (scope `drive.file`).
- Удалённые (перемещённые в корзину) файлы недоступны — возвращается `FILE_NOT_FOUND`.

## 10. POST purge (v2.0)

Безвозвратное удаление всех записей с `is_deleted = true` из всех листов и неиспользуемых файлов обложек с Google Drive. Вызывается пользователем вручную через настройки. Подтверждение получается на стороне клиента до вызова API.

**Request**

```
POST {WEB_APP_URL}
```

```json
{
  "action": "purge",
  "confirm": true
}
```

**Request — поля**

| **Поле** | **Тип** | **Обяз.** | **Описание** |
|----------|---------|-----------|--------------|
| `action` | string | Да | Значение: `"purge"` |
| `confirm` | boolean | Да | Должен быть `true`. Защита от случайного вызова. |

**Response (200)**

```json
{
  "ok": true,
  "purged": {
    "tasks": 12,
    "goals": 2,
    "contexts": 1,
    "categories": 0,
    "checklist_items": 8,
    "cover_files": 1
  }
}
```

### 9.1. Логика

1. Проверить `confirm = true`. Если нет — вернуть ошибку `CONFIRMATION_REQUIRED`.
2. Для каждого листа (`Tasks`, `Goals`, `Contexts`, `Categories`, `Checklist_Items`): найти все строки с `is_deleted = true`, удалить их физически, подсчитать количество.
3. Собрать все `cover_file_id` из удалённых целей.
4. Для каждого `cover_file_id` — проверить, есть ли ссылки из оставшихся активных целей. Если ссылок 0 — удалить файл с Google Drive.
5. Вернуть количество удалённых записей по каждой сущности.

## 11. Обработка ошибок

### 10.1. Формат ошибки

```json
{
  "ok": false,
  "error": "NOT_INITIALIZED",
  "message": "Вызовите init перед использованием API"
}
```

### 10.2. Коды ошибок

| **Код** | **Описание** |
|---------|--------------|
| `INVALID_ACTION` | Неизвестный `action` в запросе |
| `NOT_INITIALIZED` | Структура данных не создана. Необходимо вызвать `init` |
| `INVALID_PAYLOAD` | Невалидные данные в запросе (пропущены обязательные поля, неверный тип) |
| `CONFIRMATION_REQUIRED` | Поле `confirm` не передано или равно `false` (purge) |
| `FILE_TOO_LARGE` | Размер файла обложки превышает 2 МБ |
| `FILE_NOT_FOUND` | Файл обложки не найден на Google Drive |
| `INTERNAL_ERROR` | Внутренняя ошибка сервера |

## 12. Поток синхронизации

### 11.1. Первый запуск

1. Пользователь вставляет URL Web App в настройки PWA.
2. Клиент вызывает `ping` — проверяет доступность.
3. Если `initialized = false` — клиент вызывает `init`.
4. Клиент вызывает `pull` с версиями `0` — получает настройки по умолчанию (пустые данные).

### 11.2. Регулярная синхронизация

1. При открытии приложения (online) — `pull` (получить изменения с других устройств).
2. После локальных изменений — `push` (отправить на сервер).
3. Периодически (каждые 5 минут) — `pull` для подхвата изменений.
4. Офлайн: клиент накапливает изменения в IndexedDB, при восстановлении сети — `push`, затем `pull`.

### 11.3. Оптимизация

- **Debounce push:** не отправлять push после каждого изменения, а собирать пачку за 15 секунд после последнего действия
- **Batch read:** сервер читает все листы за один вызов `getDataRange()` вместо поячеечного чтения
- **Batch write:** сервер собирает все изменения и применяет через `setValues()` за минимальное количество вызовов `SpreadsheetApp`

### 11.4. Лимиты Apps Script

- Максимальное время выполнения скрипта: **6 минут**
- Максимальный размер POST-тела: **~50 МБ** (но рекомендуется до 2 МБ для обложек)
- Квота вызовов: **20 000 / день** (при sync-подходе тратится 10–30 вызовов в день)
- Холодный старт: **~1–2 секунды** на каждый HTTP-запрос
