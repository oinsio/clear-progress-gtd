# Clear Progress

**GTD-приложение для доведения дел до конца**

## СТРУКТУРА ДАННЫХ

Google Sheets Schema

Версия 1.0 | Март 2026

---

## 1. Общие принципы

### 1.1. Хранилище

Данные хранятся на двух уровнях: серверное хранилище (Google Sheets в аккаунте пользователя) и клиентский кэш (IndexedDB через Dexie.js). Структура таблиц в IndexedDB полностью повторяет структуру листов Google Sheets.

### 1.2. Идентификаторы

Все сущности используют UUID v4 в качестве первичного ключа. UUID генерируется на клиенте (через `crypto.randomUUID()`), что позволяет создавать записи офлайн без конфликтов при синхронизации. Формат: 8-4-4-4-12 символов (36 символов с дефисами).

### 1.3. Удаление записей

Применяется мягкое удаление (soft delete): поле `is_deleted` устанавливается в `TRUE`, строка остаётся в таблице. Это обеспечивает корректную синхронизацию и возможность восстановления. Клиент фильтрует удалённые записи при отображении.

### 1.4. Синхронизация

Каждая запись содержит поле `version` (целое число, начиная с 1). При каждом изменении записи `version` увеличивается на 1. Клиент сравнивает локальную и серверную версии для определения направления синхронизации. Поля `updated_at` и `created_at` хранят временные метки в формате ISO 8601 (UTC).

### 1.5. Обложки целей

Изображения-обложки загружаются через Google Apps Script в подпапку `Covers` внутри папки `Clear Progress` на Google Drive пользователя. В Google Sheets хранится только file ID файла на Google Drive. Клиент получает изображение через API `get_cover` и кэширует его в IndexedDB.

### 1.6. Файловая структура на Google Drive

При первом запуске Apps Script автоматически создаёт следующую структуру в корне Google Drive пользователя:

- **Clear Progress/** — корневая папка приложения
- **Clear Progress/Clear Progress Data.gsheet** — основной файл данных (6 листов)
- **Clear Progress/Covers/** — подпапка для изображений-обложек целей

Скрипт сохраняет ID папки и файла в `PropertiesService` для быстрого доступа при последующих вызовах.

### 1.7. Безопасность и доступ

Скрипт использует ограниченный OAuth-scope `drive.file` вместо полного доступа к Google Drive. Этот scope даёт доступ только к файлам, которые скрипт сам создал. Остальные файлы пользователя недоступны скрипту даже теоретически.

Техническая реализация: в манифесте `appsscript.json` указывается scope `https://www.googleapis.com/auth/drive.file`. Вместо `DriveApp` используется Drive API (Advanced Service), который поддерживает гранулярные scope. Аналогично, для Google Sheets используется scope `https://www.googleapis.com/auth/spreadsheets`.

При авторизации пользователь видит чёткое описание: «Доступ только к файлам, созданным этим приложением» — это вызывает больше доверия, чем запрос полного доступа к Drive.

### 1.8. Структура листов Google Sheets

Один файл Google Sheets содержит 6 листов (sheets):

- **Tasks** — задачи (основная таблица)
- **Goals** — цели
- **Contexts** — контексты
- **Categories** — категории
- **Checklist_Items** — пункты чеклистов (v1.1)
- **Settings** — настройки пользователя

Первая строка каждого листа — заголовки колонок. Данные начинаются со второй строки.

## 2. Лист Tasks

Основная таблица приложения. Каждая строка — одна задача. Связи с целями, контекстами и категориями — через UUID-ссылки.

| **Колонка** | **Тип** | **Обяз.** | **Описание** |
|---|---|---|---|
| **id** | string | Да | Первичный ключ. UUID v4, генерируется на клиенте. |
| **title** | string | Да | Название задачи. |
| **notes** | string | Нет | Текстовые заметки / дополнительная информация. |
| **box** | enum | Да | Коробка задачи. Допустимые значения: `today`, `week`, `later`, `inbox`. |
| **goal_id** | string | Нет | UUID цели из листа Goals. Пусто, если задача не привязана к цели. |
| **context_id** | string | Нет | UUID контекста из листа Contexts. |
| **category_id** | string | Нет | UUID категории из листа Categories. |
| **is_completed** | boolean | Да | Флаг завершения. По умолчанию: `FALSE`. |
| **completed_at** | datetime | Нет | Дата и время завершения (ISO 8601 UTC). Заполняется при `is_completed = TRUE`. |
| **repeat_rule** | string | Нет | (v1.1) Правило повторения. Формат: `daily`, `weekly`, `monthly`, `weekdays:1,3,5`, `custom:N` (N дней). Пусто для одноразовых задач. |
| **sort_order** | number | Да | Порядок сортировки внутри коробки. Целое число, меньшее = выше в списке. |
| **is_deleted** | boolean | Да | Флаг мягкого удаления. По умолчанию: `FALSE`. |
| **created_at** | datetime | Да | Дата создания записи (ISO 8601 UTC). |
| **updated_at** | datetime | Да | Дата последнего изменения (ISO 8601 UTC). |
| **version** | number | Да | Счётчик версии для синхронизации. Начинается с 1, +1 при каждом изменении. |

## 7. Лист Settings

Настройки пользователя в формате ключ-значение. Три колонки: `key` (string), `value` (string), `updated_at` (datetime).

### 7.1. Схема листа

| **Колонка** | **Тип** | **Обяз.** | **Описание** |
|---|---|---|---|
| **key** | string | Да | Уникальный ключ настройки. См. таблицу ключей ниже. |
| **value** | string | Да | Значение настройки. Тип зависит от ключа (хранится как строка). |
| **updated_at** | datetime | Да | Дата последнего изменения (ISO 8601 UTC). |

### 7.2. Реестр ключей

| **Ключ (key)** | **Описание и допустимые значения** |
|---|---|
| **default_box** | Коробка по умолчанию для новых задач. Допустимые значения: `today`, `week`, `later`, `inbox`. По умолчанию: `inbox`. |
| **accent_color** | Акцентный цвет интерфейса. Предустановленные: `green`, `orange`, `purple`, `yellow`, `crimson`. Также поддерживается произвольный HEX-код цвета в формате `#RRGGBB` (например, `#FF5722`). По умолчанию: `green`. |
| **creation_fields** | (v1.1) Поля, отображаемые при создании задачи. JSON-массив ключей: `["goal","context","category","notes","checklist"]`. По умолчанию: все поля. |
| **quick_property** | (v1.1) Свойство для быстрого доступа из списка задач. Допустимые: `goal`, `context`, `category`. По умолчанию: `goal`. |
| **menu_always_visible** | (v1.1) Меню всегда видимо. Допустимые: `true`, `false`. По умолчанию: `false`. |
| **menu_items** | (v1.1) Видимые пункты меню. JSON-массив: `["inbox","contexts","categories","goals","tasks","completed"]`. По умолчанию: все пункты. |
| **inbox_fields** | (v1.1) Поля, запрашиваемые при переносе из Inbox. JSON-массив: `["context","category","goal","repeat"]`. По умолчанию: все поля. |
| **focus_entity** | (v1.1) Закреплённая сущность в режиме Фокус. Формат: `тип:uuid` (например, `goal:abc-123` или `context:def-456`). Пусто = режим выключен. |

## 8. Связи между листами

Схема связей между сущностями (через UUID-ссылки):

- **Tasks.goal_id** → Goals.id — задача принадлежит цели (0..1 : N)
- **Tasks.context_id** → Contexts.id — задача выполняется в контексте (0..1 : N)
- **Tasks.category_id** → Categories.id — задача относится к категории (0..1 : N)
- **Checklist_Items.task_id** → Tasks.id — пункт чеклиста принадлежит задаче (1 : N)

Все связи опциональны (кроме `Checklist_Items.task_id`). Задача может существовать без привязки к цели, контексту или категории. При мягком удалении цели/контекста/категории ссылки в Tasks сохраняются, но клиент отображает их как «без привязки».

## 9. Ограничения Google Sheets

- Максимум 10 000 000 ячеек на файл (~660 000 задач при 15 колонках)
- Максимум 50 000 символов на ячейку (достаточно для заметок)
- Apps Script: 6 минут максимальное время выполнения скрипта
- Apps Script: 20 000 вызовов API в день (бесплатный аккаунт)
- Google Drive: 15 ГБ общее хранилище (бесплатный аккаунт, включая обложки целей)

Для персонального GTD-приложения эти лимиты более чем достаточны. Даже при активном использовании (1000+ задач, 50+ целей) запас по лимитам остаётся многократным.
